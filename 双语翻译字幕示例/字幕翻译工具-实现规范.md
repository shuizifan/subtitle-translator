# 在线字幕翻译工具 — 实现规范与技术栈（草案 v0.2）

> 本文档用于指导 Claude Code 进行开发。标注「📌 决策」的地方是有取舍的设计选择，可以推翻；标注「⚠️ 坑」的地方是实战中容易踩、建议照做的点。

---

## 0. 文档状态与已锁定决策

- 第一版（v1）范围：仅处理字幕文件；不处理视频、不做语音识别、不做转码、不做内嵌。
- **已锁定（v0.2）**：
  - 部署：**公网部署**，宝塔面板**「Node 项目」**托管（见 §1）。
  - 技术栈：**Next.js + TypeScript**，唯一后端是单个转发路由（见 §2）。
  - API：**只做 OpenAI 兼容协议**一种（见 §9）。
  - 密钥：UI 手填、存 localStorage、经转发路由穿透；不写进源码/JS 包；强制 HTTPS（见 §9）。
  - 双语布局：参考字幕实测为**「双条目同时间轴」**式；导出器**两种布局都支持**，默认沿用此式（见 §7、附录 A）。

## 1. 整体定位与部署形态

这是一个**公网部署的 Web 应用**，通过**宝塔面板「Node 项目」**托管（PM2 管理进程），用 Nginx 反代 + Let's Encrypt 上 HTTPS，并用宝塔的网站密码（HTTP Basic Auth）把整站挡在登录后。

它由两部分组成：

- **前端 SPA**：负责上传、解析、预览、配置、样式设置、导出、下载等几乎所有逻辑。
- **薄后端（LLM 转发路由）**：一个**无状态**的转发层，只做一件事——把翻译请求转发给用户配置的 OpenAI 兼容 API。**这是唯一的后端**，在 Next.js 里就是 `app/api/translate/route.ts` 一个文件。

> 心智模型：**本质是「纯前端 + 一个转发文件」**。没有数据库、没有账号体系，符合「纯前端那种」的设想；加这一层转发只是为了解决 CORS 和密钥暴露。

### 📌 决策：为什么必须有这层转发（哪怕只想要纯前端）

浏览器直连 OpenAI 兼容 API 在公网部署下有两个硬问题：

1. **CORS 不可控**。「各厂商都适配 OpenAI 协议」指的是**接口格式**，不等于**允许浏览器跨域直连**。OpenAI 本身，以及 DeepSeek、Moonshot(Kimi)、智谱等很多兼容端点都不返回 CORS 头，浏览器直连会被拦死。
2. **密钥暴露**。若做成纯静态站点把 key 打进 JS 包，任何过了宝塔密码的人都能从前端源码里读出 key。

转发路由（`浏览器 → /api/translate → 模型 API`）同时解决这两点，且无状态、部署简单。

> 例外：若你**只**用 Ollama 本地模型或 OpenRouter 这类明确允许浏览器直连的服务，理论上可纯前端、不要转发。但公网 + 多供应商场景下不推荐——默认保留转发层。

## 2. 技术栈

### 📌 推荐栈（主选）

| 层 | 选型 | 理由 |
|---|---|---|
| 框架 | **Next.js（App Router）+ TypeScript** | 前端与代理路由同仓库、一次部署；Claude Code 对其支持成熟 |
| UI | **React + Tailwind CSS + shadcn/ui** | 组件齐全、搭界面快、风格统一 |
| 状态管理 | **Zustand** | 轻量，适合管理「任务 / 工作流」这种贯穿全程的状态 |
| 拖拽上传 | **react-dropzone** | 拖拽 + 点选一把梭 |
| 编码探测 | **jschardet** + **iconv-lite** | 处理非 UTF-8 字幕（见 §5） |
| LLM 调用 | `fetch` 直连 OpenAI 兼容端点，或 `openai` npm SDK 指向自定义 `baseURL` | 一套客户端覆盖绝大多数供应商 |
| 并发控制 | **p-limit** | 限制同时进行的翻译批次数 |
| 测试 | **Vitest** | 给解析器/序列化器/对齐逻辑写单测，防回归 |
| 部署 | **宝塔面板「Node 项目」+ PM2**；Nginx 反代 + HTTPS + 网站密码 | 公网自托管；构建后 PM2 跑 `next start`（或 standalone 产物）|

界面保持简单：主要是单一工作流页面 + 一个 API 代理路由，尽量用 Client Component，不必过度使用 Server Component。

### 📌 决策：为什么是 Node 项目 + Next.js（而非 Python / 纯静态 HTML 项目）

- **不用 Python 项目**：本工具重前端（上传、预览、样式编辑器都是 UI），后端只有「转发翻译」一件事。用 Python 会导致前后端两种语言两套工程，对 vibe coding 添乱，Python 在此无优势。
- **不用纯静态「HTML 项目」**：见 §1，公网下会撞 CORS + 密钥暴露两堵墙。
- **用 Node 项目 + Next.js**：前端 + 唯一的转发路由同仓库、一次构建一次部署；宝塔 Node 项目对 PM2 托管支持成熟。

### 宝塔部署要点

1. 宝塔新建「Node 项目」，指向仓库目录；启动命令用 `npm run build` 后由 PM2 跑 `npm start`（Next.js 默认 3000 端口）。
2. 宝塔「网站/反向代理」把域名反代到该端口，开 SSL（Let's Encrypt 一键）。
3. 站点设置里加**网站密码（Basic Auth）**，挡住整站。

### 备选栈（更轻）

不想用 Next.js 时：**Vite + React**（纯静态前端）+ **Hono/Express**（极轻 Node 服务，既转发又托管静态文件），同样跑在宝塔 Node 项目里、单进程。概念上更贴近「纯前端 + 一个转发」，但要自己接静态托管，比 Next.js 多一点活。

### 📌 决策：编排放在客户端

整条字幕翻译可能跑几分钟（几百次 API 调用）。**由客户端循环驱动批次**、逐批调用 `/api/translate`、自己更新进度——这样进度天然、可断点续传，后端保持无状态。不采用服务端长任务 + SSE 流式的复杂方案。

## 3. 核心架构：公共中间模型

这是整个项目最重要的设计决策。

```
上传文件
   │
   ▼
[Parser]  任意格式 → 公共模型      (srt/ass/vtt/lrc 各一个)
   │
   ▼
SubtitleDocument（时间统一为毫秒，格式无关）
   │
   ▼
[Translator]  只认公共模型，完全不关心格式
   │
   ▼
SubtitleDocument（带译文）
   │
   ▼
[Serializer]  公共模型 → 任意格式 + 双语组装 + 样式  (各一个)
   │
   ▼
导出文件
```

### 📌 决策：好处

- 新增格式 = 加一对 parser/serializer，**引擎和 UI 一行不改**。
- 翻译、双语、样式逻辑全部写一次，对所有格式生效。
- 时间统一毫秒，避免各格式时间表示（`,` vs `.`、帧 vs 毫秒）污染核心逻辑。

## 5. 字幕上传、解析与编码处理

### 上传（对应需求第四节）

- 拖拽上传 + 点击选择（react-dropzone 一次满足）。
- **类型白名单**：`.srt / .ass / .vtt / .lrc`，同时校验扩展名与内容特征（不要只信扩展名）。
- **大小限制**：字幕文件很小，建议上限 **5 MB**（即便超长字幕也罕有超过几百 KB）。
- 上传后生成**唯一任务 ID**（客户端 UUID 即可，用于组织内存/localStorage 中的任务状态，v1 不需要服务端持久化）。
- 解析后展示：**预览**（前若干条 + 可滚动）、**总条数**。
- 失败时给出**明确错误提示**（类型不符 / 超限 / 编码无法识别 / 解析失败的具体行）。

### ⚠️ 坑：编码探测必做

SRT 在中文/日文场景下经常**不是 UTF-8**：GBK、Big5、Shift-JIS、UTF-16，还分带不带 BOM。直接按 UTF-8 读会乱码。

流程：读为字节 → `jschardet` 探测编码 → `iconv-lite` 转 UTF-8 → 去除 BOM → 再交给 parser。探测置信度低时，提示用户手动选择编码。

### ⚠️ 坑：解析器要容错

真实字幕文件常有：多余空行、时间轴格式不标准、序号跳号、最后一条无空行结尾、`\r\n` 与 `\n` 混用、HTML/ASS 标签残留。解析器要尽量容错，并在无法解析的行给出可定位的错误（第几条、原始文本）。

### ✅ 建议：先把「往返一致」测过

写完 SRT parser + serializer 后，第一件事是写单测：`serialize(parse(x))` 应与 `x` 语义等价（时间、文本、条数一致）。核心稳了再往上堆功能。

## 6. 翻译引擎（最难、最该先做对的部分）

引擎只接收公共模型，输出带译文的公共模型。

### ⚠️ 坑（头号风险）：行对齐

把多条字幕打包发给模型时，模型经常**合并或拆分行**，导致译文条数 ≠ 原文条数，时间轴随之错位。这是字幕机翻最常见的崩坏。

📌 决策——**用带 ID 的结构化输入输出 + 条数校验 + 重试**强约束：

1. 每条字幕带稳定数字 ID 发出（推荐 JSON：`[{"id":1,"text":"..."}, ...]`）。
2. 要求模型**只**返回对应结构（如以 ID 为键的 JSON），严禁合并/拆分。
3. 解析返回后**校验 ID 完整性**：缺哪几条，**只对缺失项重试**（可缩小批量、加严提示）。
4. 多次重试仍缺失的，标记为「未翻译」并在 UI 提示，绝不静默错位。

> 备选 I/O 格式：编号纯文本（`1. ...`）token 更省但解析更脆（文本里含数字/换行会干扰）。综合可靠性优先选 JSON + 严格校验 + 重试。

### 批处理

- 每批 N 条（建议默认 **20–40 条**，可在「高级参数」里调）。批越大越省请求、越有上下文，但越易触发对齐问题且单次失败损失大。
- 可选携带**前文几条**作为上下文（仅供参考、不翻译），提升连贯性（代词、称谓、专有名词）。

### 并发、限流、重试

- 用 `p-limit` 控制并发批次（建议默认 **3–5**），尊重供应商限流。
- 对 `429 / 5xx` 做**指数退避重试**。

### 进度（对应需求第六步）

- 进度 = 已完成批次 / 总批次。客户端编排时天然可得，实时更新 UI。
- 支持**取消**与**断点续传**（已完成批次的译文保留在公共模型里，失败/中断后从未完成处继续）。

### ⚠️ 坑：格式标签保护

SRT 可能含 `<i> <b> <font color>`，ASS 含 `{\an8}` 等。📌 决策：翻译前**抽离标签、翻译纯文本、译完按位置回填**，比依赖模型「请保留标签」更可靠。

### 提示词要点（`prompt.ts`）

系统提示需明确：你是专业字幕译者；从 `{源语言}` 译到 `{目标语言}`；**逐条对应、不得合并或拆分**；输出严格遵循约定的带 ID 结构；译文要符合字幕语境（口语化、简洁、单行不过长）。允许用户追加自定义风格指令（语气、领域术语倾向）。

## 7. 双语生成与样式

> 已根据用户参考字幕实测确定布局，见附录 A。

### 双语组装（`bilingual/`）

公共模型里每个逻辑 cue 同时含原文与译文；**布局由序列化器决定**，模型本身不变。SRT 双语提供两种可切换布局：

**布局一：双条目同时间轴（默认，匹配用户参考字幕）**
每个 cue 拆成两条独立 SRT 条目、共用同一时间轴，序号连续编。
```
1
00:00:01,000 --> 00:00:04,000
Original line (English)

2
00:00:01,000 --> 00:00:04,000
译文（中文行）
```
- 靠播放器把两条「同时段重叠」字幕叠显，产物接近常见机翻双语工具。
- ⚠️ 注意：两条同时间轴字幕的**上下叠放顺序由播放器决定**，文件层面不完全可控。

**布局二：单条目双行（更紧凑、顺序确定）**
```
1
00:00:01,000 --> 00:00:04,000
Original line (English)
译文（中文行）
```
- 上下顺序在文件里就钉死、各播放器渲染更一致、条数减半。

可配置项：布局（双条目 / 单条目双行）、**语言顺序**（原文在前 / 译文在前）、**导出类型**（双语 / 仅译文，对应需求第十步）。

### ⚠️ 坑：保住每个 cue 内的行结构

参考字幕里有一处机翻把英文两行对白 `-I'm coming! / -We got him!` 压成中文单行 `——我来了！——我们抓住他了！`。引擎需**逐行保译、保留 cue 内换行**，否则双语对照会错位。结合 §6 的对齐校验一起处理。

### 样式（`styling/`）

### ⚠️ 坑：SRT 样式能力很弱

SRT 只支持有限的类 HTML 标签（`<font color> <i> <b>`），且**各播放器/媒体服务器渲染不一致**（Plex/Jellyfin/Emby 对内嵌样式的支持各有取舍）。真正的样式控制（字体、字号、描边、阴影、位置、双语分色）是 **ASS 的能力**。

📌 决策：
- **v1（SRT）样式范围**：仅布局 + 语言顺序（用户参考字幕本身也未做任何颜色/字号区分，只保留了 `<i>` 斜体）。SRT 的颜色 `<font>` 各播放器支持不一，作为可选项、默认关。
- **完整样式留给 ASS 阶段**：届时用 ASS Styles + `\N` 换行，或两条不同 Style 的 Dialogue，分别控制两种语言的字体/字号/颜色/位置（如原文小而暗、译文大而亮）。

样式参数（数据结构先按 ASS 能力设计，SRT 阶段只取子集生效）：每语言的字体、字号、主色/描边色、描边/阴影、位置（顶/底）、行序。提供**预设**（对应需求第八步「有预设」）。

## 8. 导出与命名（媒体服务器识别）

媒体服务器靠**文件名后缀**识别字幕语言，命名是硬需求。

### 命名规则

通行格式：`{基础名}.{语言}.{标记}.{扩展名}`，其中基础名通常与视频同名。

- **语言代码**：提供选项 `zh` / `zh-CN` / `zh-Hans` / `chi` / `zho`（ISO 639-1 / 639-2，可带地区/字形）。
- **标记（可选）**：`.default`（默认轨）、`.forced`（强制字幕）、`.sdh`（听障字幕）。
- ⚠️ 坑：**双语没有标准语言码**。常见做法是直接用目标语言码（如 `Movie.2024.zh.srt`）；像 `zh-en` 这类自造标记媒体服务器多半识别不了、会当作未知/自定义轨道标题。建议默认用目标语言码，并在 UI 解释清楚。

### 文件名模板（对应需求第九步「有预设」）

提供变量化模板，如：`{title}.{lang}.srt`、`{title}.{lang}.default.srt`，变量含 `{title}` `{lang}` `{type}`（bilingual/translated）等。提供几个**预设**直接套用。

### 导出

让用户选择：双语 / 仅译文；选语言码、标记；预览最终文件名；一键下载。

## 9. API 配置（对应需求第三步）

### 📌 决策：只做 OpenAI 兼容协议（v1 唯一形态）

一套 `/v1/chat/completions` 客户端即可覆盖 OpenAI、DeepSeek、Moonshot、智谱、OpenRouter、Ollama 等绝大多数供应商，不为各家做特化。

用户可配置：
- **Base URL**（自定义端点）、**API Key**、**模型名**。
- 可选：temperature、max tokens、batch size、并发数（放「高级」）。
- 翻译参数：源语言（或自动检测）、目标语言、可选自定义风格提示、（后续）术语表。

可选提供一两个常见端点占位提示，但不做复杂的供应商预设体系。

提供**「测试连接」**按钮：翻译前先校验端点/密钥/模型可用。

### 📌 决策：密钥处理（公网部署，已锁定）

- **手填 + localStorage**：UI 设置项里用户手动粘贴 Base URL / Key / 模型名，存浏览器 localStorage；项目搭好前留空。这就是「纯前端那种」的手填体验。
- **经转发路由穿透，不进 JS 包**：请求走 `浏览器 → /api/translate → 模型 API`，key 在请求体/头里穿过服务端、**不落库、不写进源码、不打进前端包**。既绕开 CORS，又避免静态打包暴露 key。
- **整站挡在登录后**：宝塔网站密码（Basic Auth）+ **强制 HTTPS**（Let's Encrypt）。公网 + 带 key + 密码登录，没 HTTPS 不可接受。
- **残留风险**：key 存在用户输入它的那个浏览器里——对个人/小范围 + 密码 + HTTPS 可接受。
- **后续可选更彻底方案**：key 放服务端 `.env`，UI 连填都不用填，浏览器全程不接触 key。

> ⚠️ 不要把 key 硬编码进源码——会进 git、进构建产物、进前端包，比 localStorage 更不安全。

## 10. 公共数据模型（TypeScript 类型草案）

```typescript
export type SubtitleFormat = 'srt' | 'ass' | 'vtt' | 'lrc';

export interface SubtitleEntry {
  id: number;            // 稳定顺序索引，贯穿翻译全程
  start: number;         // 毫秒
  end: number;           // 毫秒
  originalText: string;  // 已抽离样式标签的纯文本
  translatedText?: string;
  tags?: InlineTag[];    // 抽离出的格式标签及其位置，用于回填
}

export interface SubtitleDocument {
  taskId: string;            // UUID
  sourceFormat: SubtitleFormat;
  entries: SubtitleEntry[];
  meta: Record<string, unknown>; // 保存 ASS 头/Styles/VTT 头等，用于无损往返
}

export interface InlineTag {
  raw: string;     // 原始标签，如 "<i>" 或 "{\\an8}"
  offset: number;  // 在纯文本中的插入位置
}
```

`meta` 用来无损保留格式专属信息（尤其 ASS 的 `[Script Info]` 与 `[V4+ Styles]`），保证「解析—序列化」往返不丢东西。

## 11. 错误处理与边界情况清单

| 场景 | 处理 |
|---|---|
| 文件类型不在白名单 | 明确提示支持的格式 |
| 超过大小上限 | 提示上限值 |
| 编码无法可靠探测 | 让用户手动选编码 |
| 某条字幕解析失败 | 定位到第几条 + 原始文本 |
| 翻译返回条数不匹配 | 仅重试缺失项；多次失败标记「未翻译」 |
| 网络/限流（429/5xx） | 指数退避重试 |
| 翻译中途取消/失败 | 保留已完成译文，可续传 |
| 空字幕/纯标记行 | 跳过或原样保留，不送翻译 |
| 自定义 Base URL 不通 | 「测试连接」拦在翻译之前 |

## 13. 建议的开发顺序（给 Claude Code）

1. **搭骨架** + 定义 `core/model` 公共类型。
2. **SRT parser + serializer + Vitest 单测**，先过「往返一致」。
3. **上传 + 预览 UI**（先看得到解析结果与总条数）。
4. **API 配置界面 + 测试连接**。
5. **翻译引擎**：单批 → 批处理 → 并发 → 对齐校验 → 重试 → 进度（这是项目重心，单独写测）。
6. **双语组装 + 译文预览**。
7. **样式设置**（v1 范围）。
8. **导出命名 + 下载**。
9. **宝塔 Node 项目部署**：构建 + PM2 + Nginx 反代 + HTTPS + 网站密码（见 §2「宝塔部署要点」）。

> 原则：先把 `core/`（解析 + 引擎）这块纯逻辑用测试钉死，再往上长 UI。字幕机翻的崩坏几乎都集中在「编码」和「行对齐」两处，这两处有测试守着，后面会轻松很多。
