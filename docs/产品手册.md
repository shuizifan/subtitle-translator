# 在线字幕翻译工具 — 产品手册（维护与交接版）

> 版本：v1（未发行前版本号统一为 1，见 `package.json` `"version": "1.0.0"`）
> 读者：接手维护、二次开发、部署运维的工程师
> 配套文档：[使用说明](使用说明.md)（终端用户操作）、[实现规范](../双语翻译字幕示例/字幕翻译工具-实现规范.md)（最初设计依据）

本手册的目标：让任何人不依赖原作者也能读懂结构、定位代码、安全改动、顺利部署。

---

## 1. 项目速览

| 项 | 内容 |
|---|---|
| 是什么 | 公网部署的 Web 应用：上传字幕 → 调用用户自备的 OpenAI 兼容大模型翻译 → 生成双语字幕 → 按媒体服务器命名导出 |
| 范围 | v1 仅处理**字幕文本**（SRT），不处理视频/语音识别/转码/内嵌 |
| 技术栈 | Next.js 15（App Router）+ TypeScript + React 18 + Tailwind CSS 3 + Zustand 5 |
| 后端 | **唯一**一个无状态转发路由 `src/app/api/translate/route.ts`；无数据库、无账号 |
| 模型协议 | 只做 OpenAI 兼容 `/v1/chat/completions` |
| 密钥 | 用户 UI 手填 → 浏览器 localStorage → 经转发路由穿透；不进源码/前端包 |
| 测试 | Vitest，6 个文件 / 35 用例，覆盖 core 纯逻辑 |
| 部署 | 宝塔「Node 项目」+ PM2 + Nginx 反代 + HTTPS + 网站密码 |
| 代码量 | `src/` 约 2.5k 行 |

**心智模型**：本质是「纯前端 + 一个转发文件」。前端干所有活（上传、解析、预览/编辑、翻译编排、双语组装、导出），后端那一个文件只为绕开浏览器 CORS 与避免密钥暴露而存在。

## 2. 开发者快速上手

```bash
npm install
npm run dev      # http://localhost:3000，热更新
npm test         # 跑 core 单测（必须全绿再提交）
```

- `npm run build` 会做全量类型检查，提交前务必跑一遍。
- 提交涉及版本号时保持 `1.x`（未发行前统一为 1）。

## 3. 架构总览

```
浏览器（前端 SPA，几乎所有逻辑）
   │  上传字节
   ▼
[encoding] 探测+解码  →  [parsers/srt] 解析  →  SubtitleDocument（时间统一毫秒，格式无关）
                                                      │
                          用户配置/参数（Zustand store，持久化）
                                                      │
   客户端编排（lib/useTranslator → core/translator/engine）
   逐批 → fetch('/api/translate') ───────────────┐
                                                  ▼
                              [api/translate/route.ts]（唯一后端，无状态转发）
                                                  ▼
                                       用户的 OpenAI 兼容模型 API
                                                  │ 返回 JSON
   ◀──────────────────────────────────────────────┘
   对齐校验/缺失重试 → 回填 translatedText 到 Document
                                                      │
[bilingual] 双语组装(+合并多行/配色) → [serializers/srt] → [naming] 文件名 → 浏览器下载
```

**两条铁律**（设计核心，改动时不要破坏）：
1. **公共中间模型**：所有格式先转成 `SubtitleDocument`（时间统一毫秒）；翻译/双语/样式只认这个模型，与格式无关。新增格式 = 加一对 parser/serializer，引擎与 UI 一行不改。
2. **客户端编排**：翻译循环由浏览器驱动，后端保持无状态。这样进度天然、可取消、可断点续传。

## 5. 公共数据模型（`core/model/index.ts`）

```ts
type SubtitleFormat = 'srt' | 'ass' | 'vtt' | 'lrc';   // v1 只实现 srt

interface InlineTag { raw: string; offset: number; }   // 抽离的标签 + 在纯文本中的位置

interface SubtitleEntry {
  id: number;            // 稳定顺序索引(从1)，贯穿翻译全程，行对齐的锚
  start: number;         // 毫秒
  end: number;           // 毫秒
  originalText: string;  // 已抽离标签的纯文本（可含 cue 内 \n）
  translatedText?: string;
  tags?: InlineTag[];    // 用于翻译后回填标签
}

interface SubtitleDocument {
  taskId: string;                 // 客户端 UUID
  sourceFormat: SubtitleFormat;
  entries: SubtitleEntry[];
  meta: Record<string, unknown>;  // 存 eol、parseIssues；ASS 阶段存头/Styles
}

interface ParseIssue { line: number; raw: string; message: string; }  // 解析告警(可定位)
```

> `meta` 用于无损保留格式专属信息，保证「解析—序列化」往返不丢东西。SRT 阶段存 `eol`（换行符）与 `parseIssues`。

## 6. 状态管理（`store/index.ts`）

单一 Zustand store，`persist` 中间件持久化到 localStorage（key：`subtitle-translator`）。

### 6.1 状态字段
- **文档类**（不持久化）：`fileName / detectedEncoding / document / parseIssues / phase / progress / failedIds / docVersion`。
- **配置类**（持久化）：`apiProfiles / activeProfileId / params / style / bilingual`。

`phase`: `'idle' | 'parsed' | 'translating' | 'done'`。
`docVersion`: 单调递增计数，翻译过程中节流 `bump` 以触发表格增量重渲（见 §8.6）。

### 6.2 关键 action
- `setDocument / reset / updateTranslation(id,text)`（手动改译文，原地改 entry 并 bump）。
- `selectProfile / setSourceLang / setTargetLang`：**控制条快捷切换，即时生效**。
- `applySettings(snapshot)`：**设置弹窗「保存」时整体写回**（草稿模式，见 §9）。
- `setPhase / setProgress / setFailedIds / bumpDocVersion`。
- 辅助函数：`getActiveProfile(s)`、`newProfile(partial)`、`snapshotSettings(s)`。

### 6.3 持久化与「迁移」
- `partialize`：只持久化配置类字段（文档可能很大，不存）。
- `merge`（**重要**）：自定义**深合并**，把持久化值合并到当前默认对象上 —— 保证**新增字段在老缓存上也能取到默认值**（例如后来新增的 `systemPrompt / collapseLines / translatedLabel / style.scheme`）。这是“无版本号也能平滑升级”的关键，新增 `params/bilingual/style` 字段时**无需写迁移**，深合并会兜底。
- 改了持久化结构若想强制丢弃旧缓存，可改 `name` 或加 `version`+`migrate`。

## 8. 关键算法与「坑」

### 8.1 编码探测（`core/encoding`）
流程：字节 → 查 BOM（UTF-8/UTF-16LE/BE）→ 否则 `jschardet` 探测 → `normalizeLabel` 映射成 `TextDecoder` 标签 → 解码 → 去 BOM。
- 解码用浏览器/Node 原生 `TextDecoder`（支持 gbk/big5/shift_jis/utf-16…），**不依赖 iconv-lite 的 Node Buffer**，前后端一致、可单测。
- 置信度 `<0.6` 时 UI 提示手动选编码（`useSubtitleLoader.reloadWithEncoding`）。
- 坑：中文/日文 SRT 常不是 UTF-8（GBK/Big5/Shift-JIS/UTF-16），直接按 UTF-8 读会乱码。

### 8.2 标签保护（`core/tags`）
翻译前用正则 `TAG_RE` 抽离 `<i> </i> <b> <font…> {\anX}` 等，记录 `offset`；翻译纯文本；译完 `reinsertTags` 按 offset 回填（offset 超界则夹紧到末尾，保证整行 `<i>…</i>` 包裹这类常见情况正确还原）。比依赖模型“请保留标签”可靠。

### 8.3 行对齐（`core/translator/engine.ts`，**头号风险**）
模型常合并/拆分行导致译文条数≠原文、时间轴错位。对策：
1. 每条带稳定数字 `id` 以 JSON 数组发出。
2. 系统提示强制只返回相同 `id`、逐条对应、不得合并/拆分（`prompt.ts` 的 `OUTPUT_CONTRACT`，**用户改风格也覆盖不掉**）。
3. 解析返回（`parseTranslationResponse`，容错剥代码围栏、兼容对象形式）后**校验 id 完整性**，**只对缺失项重试**。
4. 多次重试仍缺失 → 记入 `failedIds` 标红提示，**绝不静默错位**。

### 8.4 引擎其它行为
- **并发**：内置极简 `pLimit`（避免 ESM 加载差异），按 `concurrency` 限流。
- **重试**：`429 / 5xx / 网络错误` 指数退避（`500*2^n`，上限 15s，带抖动）。
- **跳过**：`originalText.trim()===''` 的空/纯标记行不送翻译。
- **断点续传**：已有 `translatedText` 的条目跳过；中断/失败后再点「继续翻译」只补缺。
- **取消**：`AbortController`，`signal` 透传到 fetch 与 sleep。
- **进度**：`onProgress`（批次/条数/失败数）+ `onEntry`（增量回填）。

### 8.5 合并多行 & 双语布局（`core/bilingual`）
- `assemble(doc, {layout, order, collapseLines}, style?)` → `OutputCue[]`。
- `collapseLines`（默认 true）：把每种语言**内部换行压成一行**（逐行 trim、丢空行、空格连接），避免「原文 3 行 + 译文 3 行 = 6 行」的观看灾难；双语只 2 行，由播放器按宽折行。
- 布局：`dual-entry`(双轨，拆两条同时间轴) / `single-entry`(单轨，一条两行) / `translated-only`(仅译文)。
- 配色：`style.enableSrtColor` 时用 `<font color>` 给原文/译文套色（见 §8.7）。

### 8.6 表格性能（`components/SubtitleTable`，**易踩**）
字幕可达数千条，早期一次性渲染 + 每行一个 `<textarea>` 会卡。现状：
- 每行 `content-visibility:auto` + `contain-intrinsic-size`：**滚动区外的行跳过布局/绘制**。
- 译文列默认是只读 `div`，**点击某行才渲染唯一一个 textarea**（不是每行一个）。
- `Row` 用 `memo`，`onStartEdit/onCommit` 用 `useCallback` 稳定；翻译时 `bump` 节流到 500ms，只有 `translated` 变化的行才重渲。
> 改这个组件时务必保留这三点，否则大文件会重新变卡。

### 8.7 命名与样式
- `core/naming.buildExportName`：`基础名 + 描述符 + 语言码 + .srt`。
  - 描述符：仅译文 `AI{中文}`、双语 `AI{中}{英}双语`（源语言已知）/`AI{中文}双语`（auto）；可被 `translatedLabel/bilingualLabel` 覆盖。
  - `deriveBaseName` 去扩展名与旧的语言/类型尾标（`.chs/.zh/.机翻中英/AI…双语` 等）。
  - 语言映射 `nounOf/charOf/codeOf`，新增语言在此加。
- `core/styling.COLOR_SCHEMES`：6 套方案（以译文/中文为主色，原文压灰，色调偏沉稳）。`DEFAULT_STYLE` 为「不分色」。

## 9. 设置弹窗的「草稿模式」（`components/SettingsDialog`）
- 打开时 `snapshotSettings(store)` 拷一份草稿到本地 state；所有编辑只动草稿。
- **「保存」**→ `applySettings(draft)` 整体写回 store；**「关闭」/点遮罩**→ 丢弃。
- 例外：控制条上的「翻译服务/原语/目标语」是**快捷控制，即时生效**，不走草稿。
- 四个标签页：翻译服务（多配置 CRUD + 预设一键带入名称/URL/模型 + 测试连接）、翻译参数、高级·提示词、双语与导出。

## 10. 后端转发路由契约（`app/api/translate/route.ts`）
- `POST`，`runtime='nodejs'`，`dynamic='force-dynamic'`，**无状态**。
- 入参：`{ baseURL, apiKey, model, messages, temperature?, max_tokens? }`；缺 `baseURL/apiKey/model/messages` → 400。
- `buildUrl`：去尾斜杠，若未以 `/chat/completions` 结尾则补全。
- 转发：带 `Authorization: Bearer <apiKey>`，`stream:false`；**原样回传上游状态码与响应体**（前端按需解析）。
- 上游 fetch 失败 → 502。
- **key 仅穿过、不落库、不记录日志**。

## 11. 主题系统（`lib/theme.ts` + `layout.tsx` + `globals.css`）
- 三态：`system`（默认）/ `light` / `dark`，存 localStorage（key `subtitle-theme`）。
- `layout.tsx` 内联脚本在 React 注水前同步给 `<html>` 加 `dark` class，**防闪烁与 hydration 不一致**（`<html suppressHydrationWarning>`）。
- Tailwind `darkMode:'class'`；`globals.css` 的共享控件类（`.card/.input/.btn-*/.label/...`）已带 `dark:` 变体；组件内联色也成对写 `dark:`。
- 维护要点：**新加 UI 时同时写浅/深两套类**，保证文字/边框/背景对比度可读。

## 13. 扩展指南（高频维护任务）

- **加默认参数 / 改默认值**：改 `store DEFAULT_PARAMS`（必要时同步 `engine DEFAULT_ENGINE_OPTIONS`）。新增字段靠 `merge` 深合并自动兜底，无需迁移。
- **加配色方案**：往 `core/styling COLOR_SCHEMES` 追加一项（`name + enableSrtColor + original/translation.primaryColor`），设置里自动出现。
- **加服务预设**：往 `SettingsDialog SERVICE_PRESETS` 加 `{label,name,baseURL,model}`。
- **加目标/源语言**：`ControlBar` 的 `SOURCE_LANGS/TARGET_LANGS`；若要影响文件名，再在 `core/naming` 的 `LANG_NOUN/LANG_CHAR/LANG_CODE` 加映射。
- **加导出标记预设**：`SettingsDialog BilingualTab` 的 `LabelField presets`。
- **新增字幕格式（ASS/VTT/LRC）**：在 `core/parsers/` 加 `parseXxx → {document, issues}`、`core/serializers/` 加 `serializeXxx(cues)`；解析时把格式头存进 `meta`；`bilingual/serializer` 与 UI 基本不用动。上传白名单在 `useSubtitleLoader` 与 `EmptyState`/`Toolbar` 的 `accept`。
- **改提示词默认**：`core/translator/prompt DEFAULT_STYLE_PROMPT`（注意 `OUTPUT_CONTRACT` 不要动，它保证对齐）。

## 14. 安全模型
- 密钥：手填 → localStorage → 转发路由穿透；**不硬编码、不进 git、不进前端包**。
- 部署：宝塔**网站密码（Basic Auth）+ 强制 HTTPS**把整站挡在登录后。公网 + 带 key + 无 HTTPS = 不可接受。
- 残留风险：key 留在用户输入它的浏览器里——对个人/小范围 + 密码 + HTTPS 可接受。
- 后续更彻底方案：key 放服务端 `.env`，浏览器全程不接触（见实现规范 §9）。

## 15. 部署与发布
### 15.1 两种部署方式
- **源码方式**：上传代码 → `npm install && npm run build` → 宝塔「Node 项目」启动命令 `npm start`（PM2，3000 端口）。升级：拉代码 →（依赖变动时）`npm install` → `npm run build` → 重启。
- **发行版方式（推荐交接/上线）**：本机 `npm run release` 打出 `release/字幕翻译_v<版本>_<时间戳>/`（standalone 最小包，含 server.js/.next/static/public/精简 node_modules），上传后直接 `node server.js`，**无需在服务器装依赖或构建**。

两种方式都要：「反向代理」域名 → `http://127.0.0.1:3000`，开 HTTPS 并强制，站点加 Basic Auth。

### 15.2 发布流程（每次优化后）
**测试 → 本地验证 → 提交源码 → 更新本手册 → `npm run release` → 归档/部署**。
详见 [发布流程](发布流程.md)。`release/` 已 gitignore（体积大不入库）。

### 15.3 站点品牌定制
改 logo/标题/描述只动 `src/config/site.ts`（构建期固化），详见 [站点定制说明](站点定制说明.md)。

## 16. 已知限制与后续路线
**当前限制 / 技术债**
- 仅 SRT；ASS/VTT/LRC 未实现。
- 表格未做真正的虚拟滚动（用 `content-visibility` 近似），超大文件（上万行）仍可能有滚动条估高跳动。
- 标签回填对“行内多段标签 + 译文长度大变”是近似（offset 夹紧）；整行包裹场景精确。
- `p-limit` 依赖虽在 `package.json`，引擎实际用内置 `pLimit`（保留依赖无害）。
- 翻译进度/已上传文档不持久化（刷新需重新上传，配置会保留）。

**路线（沿用实现规范 §12）**：ASS 解析+完整样式（双语分色/分位）→ VTT/LRC → 术语表/一致性 → 多任务持久化 → 服务端密钥存储。
