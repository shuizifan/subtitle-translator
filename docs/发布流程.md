# 发布流程（优化 → 发版 → 交接）

每次优化项目后，按本流程产出三样东西：**更新后的源码**、**带时间戳的发行版目录**、**更新后的产品手册**。

---

## 一、发行版是什么

`npm run release` 会基于 Next.js **standalone** 产物，把「部署所需的最小文件集」复制到：

```
release/字幕翻译_v<版本>_<时间戳>/
├── server.js          # 启动入口：node server.js
├── .next/             # 服务端产物 + static 静态资源
├── public/            # 静态资源（logo 等）
├── node_modules/      # 仅运行所需的精简依赖
├── package.json
├── docs/              # 随包附带的文档（手册/使用说明/定制/本流程）
└── 启动说明.txt        # 部署/启动须知
```

特点：**无需再 `npm install` / `npm run build`**，上传即可 `node server.js` 运行。
`release/` 已加入 `.gitignore`（体积大、含构建产物，不进 git；按需归档或传输）。

---

## 二、标准优化发布流程

> 约定：未发行前版本号统一为 **1**（`package.json` 保持 `1.x`）。

1. **改代码**（实现优化/修复）。
2. **跑测试**：`npm test` —— 必须全绿（core 逻辑回归防线）。
3. **本地验证**：`npm run dev` 手测关键路径（上传 → 翻译 → 双语导出）。
4. **更新源码**：提交 git（在功能分支上 commit；如需上线再合并）。
   - 提交信息写清这次优化了什么。
5. **更新产品手册**：若改动涉及结构/默认值/模块职责/新功能，同步改 [产品手册](产品手册.md)（必要时也改 [使用说明](使用说明.md)）。
6. **打发行版**：`npm run release`
   - 自动先构建再打包，生成 `release/字幕翻译_v<版本>_<时间戳>/`。
   - 跳过构建（已构建过）：`npm run release -- --no-build`。
7. **归档/部署**：把该时间戳目录归档保存，并按需上传服务器部署。

一句话顺序：**测试 → 验证 → 提交源码 → 更新手册 → `npm run release` → 归档/部署**。

---

## 三、部署发行版（宝塔 Node 项目）

1. 把 `release/字幕翻译_v..._...` 整个目录上传到服务器，如 `/www/wwwroot/subtitle-translator`。
2. 宝塔「Node 项目」：项目目录指向该文件夹，**启动命令 `node server.js`**，用 PM2 守护。
   - 可设环境变量：`PORT=3000`、`HOSTNAME=0.0.0.0`。
3. 「反向代理」域名 → `http://127.0.0.1:3000`，开 Let's Encrypt **HTTPS** 并强制。
4. 站点「网站密码」加 **Basic Auth**，整站挡在登录后。

> 更换 logo/标题/描述属于构建期固化，发行版里改不了；需回源码改 `src/config/site.ts` 后重新 `npm run release`（见 [站点定制说明](站点定制说明.md)）。

---

## 四、版本与归档建议

- 目录名已含**版本号 + 时间戳**（精确到分钟），天然可区分多次发版。
- 如需长期留存，建议把目录压缩成 `字幕翻译_v1.0.0_YYYYMMDD-HHMM.zip` 存到对象存储/网盘，而非提交进 git。
- 回滚：保留上一份发行版目录，出问题时直接切回旧目录重启。
