# 在线托管 / 一键部署指南

本项目**不是纯静态站**：它带一个服务端转发路由 `/api/translate`（绕开浏览器 CORS、避免密钥进前端包）。
因此需要能**运行 Node / 服务端函数**的平台，不能用纯静态托管。

> 关键认知：**连 Git 仓库、让平台自己构建**，你不用上传 `node_modules`，也就没有「文件数上限」问题。
> `npm run release` 打的发行版包是给**宝塔/自有服务器**自托管用的，与下面这些托管平台无关。

---

## 0. 通用前置：把代码推到 Git

```bash
git init
git add .
git commit -m "init"
# 在 GitHub/GitLab 新建空仓库后：
git remote add origin <你的仓库地址>
git push -u origin main
```

之后在各平台「Import Git Repository」即可。框架会被自动识别为 **Next.js**，无需手填构建命令（默认 `next build`）。

---

## 1. 平台对比（按契合度）

| 平台 | 适配度 | 一句话 |
|---|---|---|
| **Vercel** ⭐ | 最佳 | Next.js 官方平台，导入仓库**零配置**，API 路由原生支持，免费 Hobby 档够用 |
| **Netlify** | 好 | 官方 Next 插件，连 Git 一键，免费档可用 |
| **Railway** | 好 | 跑常驻 Node 服务，连 Git 一键，按用量计费 |
| **Render** | 好 | 同上；免费 Web Service 会闲置休眠（首次访问冷启动慢） |
| **Zeabur / Sealos**（国内友好） | 好 | 支持 Next.js，连 Git 部署，国内访问更快 |
| **Cloudflare Pages** | 需改造 | 边缘环境，需把 `/api/translate` 改 `edge` 运行时 + `@cloudflare/next-on-pages` |

## 2. Vercel（推荐）

1. 访问 vercel.com，用 GitHub 登录。
2. **Add New → Project → 选你的仓库 → Import**。
3. 框架自动识别 Next.js，构建/输出**全部默认**，直接 **Deploy**。
4. 几分钟后得到 `https://xxx.vercel.app`，`/api/translate` 自动作为函数运行。
5. 绑定自定义域名：Project → Settings → Domains。

- 本项目无需任何环境变量（密钥由用户在前端填、经路由穿透）。
- 加访问保护见 §6。

---

## 3. Netlify

1. app.netlify.com → **Add new site → Import an existing project** → 选仓库。
2. 自动用官方 `@netlify/plugin-nextjs`，构建命令 `next build` 默认即可 → Deploy。
3. 得到 `https://xxx.netlify.app`，API 路由作为 Netlify Functions 运行。

---

## 4. Railway / Render（跑常驻 Node 服务）

适合想要「一直开着的 Node 服务器」。

**Railway**：New Project → Deploy from GitHub repo → 自动检测 Next.js → 构建 `npm run build`、启动 `npm start`。在 Settings 暴露生成的域名即可。

**Render**：New → **Web Service** → 连仓库 →
- Build Command：`npm install && npm run build`
- Start Command：`npm start`
- 免费档会在闲置后休眠，首次访问有冷启动延迟。

---

## 5. Cloudflare Pages（需要改造）

CF Pages 是**静态 + 边缘函数**，直接上传产物会撞文件数上限，且我们的路由是 Node 运行时，需改造：

1. 把 `src/app/api/translate/route.ts` 顶部的 `export const runtime = "nodejs"` 改为 `export const runtime = "edge"`（该路由只用 `fetch`，兼容边缘）。
2. 安装并用官方适配器 `@cloudflare/next-on-pages` 构建：
   - 构建命令：`npx @cloudflare/next-on-pages`
   - 输出目录：`.vercel/output/static`
3. 在 CF Pages「连接 Git 仓库」方式部署（**不要手动上传产物**），由 CF 构建，避免文件数问题。

> 如果不想折腾边缘适配，直接用 Vercel 最省心。需要我帮你做这套 CF 改造可随时说。

## 6. ⚠️ 公网部署的安全要点（务必看）

1. **不要硬编码 API Key**：本项目密钥由用户在浏览器本地填写、经路由穿透，代码里没有 key——保持如此，别写死。
2. **`/api/translate` 是开放转发代理**：它会转发到用户填的任意 Base URL。公开部署等于对外开放了一个转发端点，可能被人当跳板滥用（消耗你的额度/带宽，或 SSRF）。
   - 自托管（宝塔）时靠「网站密码 Basic Auth」挡住。
   - 放到 Vercel/Netlify/CF 时建议加一道**访问保护**：
     - Vercel：Settings → **Deployment Protection / Password Protection**（部分功能需付费档）；
     - 或加 **Basic Auth 中间件**（`middleware.ts`，对全站校验账号密码）——这套我可以帮你加；
     - 或在路由里**限制允许转发的目标域名白名单**。
3. **强制 HTTPS**：上述平台默认都是 HTTPS。

## 7. 我该选哪个？

- 只想最快上线、省心 → **Vercel**。
- 团队已在用 Netlify → **Netlify**。
- 想要常驻服务器/更可控 → **Railway**（或自有服务器 + `npm run release`）。
- 必须用 Cloudflare → 按 §5 改造（或让我代劳）。

> 想要哪个平台的「改好就能一键部署」版本（含访问口令中间件），告诉我目标平台即可。
